Agentic Systems · Guest Essay

There's a peculiar kind of amnesia that grips every new generation of technology. We declare a paradigm shift, tear up the old rulebooks, and spend years rediscovering constraints that the previous generation already mapped, named, and solved — or learned to live with.

AI engineering is deep in that phase right now. The problems showing up in production agentic systems — degraded context, confused shared state, unauditable behaviour, privilege escalation, coordination failures — are not novel. They are, almost without exception, problems that systems engineers have been solving since the 1970s. The solutions are often there. We just haven't applied them.

I've spent the past year building and running agentic systems daily: a persistent memory engine in Rust, an autonomous engineering pipeline, a task-routing system, a nightly memory maintenance process. The problems I kept hitting weren't AI problems. They were systems problems. Here's what classical engineering has to teach us.

The Analogies
01

Garbage Collection→Memory Lifecycle Management

Before garbage collection, C programmers managed memory manually. malloc to allocate, free to release. Forget to free something and it leaks. Free something still in use and you corrupt your program. The solution — automated tracking of what's live, what's dead, and periodic collection — is one of the great engineering contributions of the last century.

Now consider a persistent AI memory store running across multiple agents over months of daily use. Facts accumulate. Some are accessed constantly; some were written once in January about a project that was scrapped. All of them sit in the store with equal weight. Every agent that reads from this store is implicitly treating a stale, invalidated fact as equivalent to a freshly confirmed one.

Classical GC Problem
Memory leak — objects accumulate, never freed
Dangling pointer — reference to freed memory still used
Reference counting — track how often a value is used
Generational GC — old objects scrutinised more than young
AI Memory Equivalent
Memory bloat — stale facts accumulate indefinitely
Stale retrieval — invalidated memory still retrieved as truth
Access frequency — hot memories promoted; cold ones flagged
Confidence decay — old, low-recall memories decay first

The solution I've been running is a nightly process — memory-dreamer — that snapshots the full memory database, runs analytics across access frequency and age, promotes heavily-used memories, invalidates cold ones, and propagates confidence decay through the knowledge graph when something is marked stale. It's garbage collection for a memory palace.

Key Principle

Every memory system needs a lifecycle. Capture without collection is just structured confusion. A memory your agent confidently retrieves is implicitly more trusted than one it doesn't — so anything left in the store long enough becomes implicitly authoritative, whether it deserves to be or not.

02

Memory Safety & Ownership→Context Ownership in Shared Agent Memory

Rust's borrow checker enforces a simple rule: one owner, one mutable reference at a time. This eliminates a whole class of bugs — data races, use-after-free, double-free — by making the compiler reject code that would produce them. The elegance isn't in the constraint; it's in making the constraint unambiguous.

Now run three agents writing to a shared memory store simultaneously. Agent A records a decision about the auth module. Agent B, one second later, records a conflicting decision about the same module based on slightly older context. Agent C reads both in the same retrieval. What does it believe?

Nobody owns that memory. Nobody knows who last wrote it or why. The store has no concept of provenance. In systems terms, this is a write race on shared mutable state — one of the oldest, nastiest categories of bug in concurrent programming. 

rust analogy
// In Rust: only one writer allowed. Compiler enforces it.
let mut memory = MemoryStore::new();
let writer_a = &mut memory;  // OK
let writer_b = &mut memory;  // COMPILE ERROR — already borrowed

// In naive multi-agent systems: both just write.
// No ownership model. Last writer wins. Silently.
agent_a.write("auth module uses JWT");
agent_b.write("auth module uses sessions");
// Ground truth is now ambiguous. No error raised.

The fix borrows from both Rust and database design: writes to shared memory should carry provenance (which agent, which session, which timestamp, with what confidence). Conflicting writes should be surfaced, not silently overwritten. Retrieval logic should prefer the most-recently-confirmed version from the most-trusted source.

03

Security & Trust Boundaries→Prompt Injection & Agent Trust

SQL injection. Buffer overflows. Cross-site scripting. Every major class of security vulnerability from the past three decades follows the same structure: untrusted input reaches a context where it's interpreted as trusted instruction. The solution, every time, is the same: never conflate data with code. Sanitise at the boundary. Enforce least privilege.

Prompt injection is SQL injection for language models. An adversary embeds instructions inside content the agent will read — a webpage, an email, a document, a tool result — and the agent, lacking a principled boundary between data it is processing and instructions it should follow, executes them.

Classical Security Pattern
SQL injection — untrusted input in query context
Buffer overflow — untrusted data overruns instruction space
Privilege escalation — low-trust process gains high-trust access
Sandboxing — untrusted code runs with constrained permissions
Audit log — every privileged action traced and reviewable
AI Agent Equivalent
Prompt injection — instructions embedded in observed content
Context overflow — injected content displaces real instructions
Jailbreak / role confusion — agent tricked into acting beyond scope
Tool scope limits — agents run with constrained action sets
Action tracing — every tool call logged with rationale
The Uncomfortable Truth

We do not yet have a principled, general solution to prompt injection the way parameterised queries solved SQL injection. The closest we have is defence-in-depth: structured parsing of tool outputs, explicit trust levels on data sources, human-in-the-loop gates before irreversible actions, and agent scope restriction. It's 1998-era security thinking. We need our prepared statements moment.

The human approval gate in my eng-crew pipeline is, at its core, a security control. Before any code is executed — before the blast radius matters — a human reviews the plan. It's the AI equivalent of a firewall rule review before a deployment. Tedious to implement, critical to trust.

04

Concurrency & Race Conditions→Parallel Agent Coordination

Distributed systems engineers know that two processes updating the same resource without coordination will, eventually, corrupt it. Mutexes, semaphores, compare-and-swap, optimistic locking, event sourcing — decades of tooling exist specifically because concurrent mutation of shared state is hard to get right.

Multi-agent pipelines are distributed systems. When I run four parallel coder agents on separate subtasks in eng-crew, they each have opinions about the state of the codebase. They may modify the same files. One agent's plan may assume a function signature that another agent is in the process of changing. 

conceptual
// Classical deadlock
Thread A holds Lock X, waits for Lock Y
Thread B holds Lock Y, waits for Lock X
→ deadlock.

// Agent equivalent
Agent A: "I'll refactor the auth module once the API layer is settled"
Agent B: "I'll settle the API layer once auth module types are final"
→ both blocked. No error. Just silence and missed deadlines.

The decisions MCP server I built is a partial answer to this: it ranks tasks and routes agents based on dependencies, ensuring agents work in an order that respects what each one needs from the others. It's a scheduler. Distributed systems have had schedulers for a long time. We're reinventing them for agents.

05

Cache Invalidation→Knowledge Graph Staleness

Phil Karlton's famous quip — that naming things and cache invalidation are the two hardest problems in computer science — has aged well. The reason cache invalidation is hard isn't the invalidation itself; it's the dependency graph. Invalidate fact A, and everything that was derived from fact A may now be wrong. But you have to know what depended on A to know what else needs invalidating.

In a persistent agent memory store, this problem is structural. A memory about a module's architecture is linked to memories about its API surface, its test strategy, its deployment configuration. If the architecture changes and that root memory is invalidated, the downstream memories don't automatically inherit the invalidation. They become quietly wrong — confidently cached, subtly stale.

The answer is a knowledge graph with typed edges and a confidence propagation model — so that invalidating a parent node decays the confidence of everything downstream, weighted by edge strength and traversal distance.

The rusty-mempalace architecture supports exactly this: memories are nodes, relationships are typed edges, and the nightly maintenance process propagates confidence decay through the graph when something upstream is invalidated. It's cache invalidation. It just took a while to recognise it as such.

06

Process Isolation & Sandboxing→Agent Scope & Blast Radius

Modern operating systems give each process its own address space. A buggy process can crash itself without corrupting memory used by other processes. It can be killed without bringing down the system. The blast radius of a failure is contained by design.

Agentic systems often have the opposite property: a single agent with broad tool access and no scope limits can read and write across the entire environment it operates in. A misunderstanding, a bad plan, a prompt injection attack — any of these can produce actions whose blast radius is the entire project, codebase, or data store.

Scope restriction is the agent equivalent of process isolation: the architect agent may think broadly but should not execute. The coder agent executes but only on files within its assigned subtask. The reviewer reads broadly but writes nothing. Different agents, different permission sets, hard limits enforced — not by convention, but by the infrastructure.

Anti-Pattern to Avoid

Giving every agent full tool access because it's easier to configure. This is the equivalent of running everything as root. It works until it doesn't, and when it doesn't, the failure is total. Design the minimal permission set each agent needs for its specific role. Treat broad access as a code smell.

07

Logging & Observability→Agent Action Tracing

The observability movement in software engineering made a simple argument: a system you can't inspect is a system you can't trust in production. Structured logs, distributed tracing, metrics, alerting — these aren't features you add after deployment. They're the difference between a system you operate and one you just hope works.

Most agentic systems today have poor observability by default. You know what task you gave an agent. You may know what output it produced. The reasoning chain in between — what it retrieved, what it considered, what alternatives it discarded, which tool call triggered which downstream action — is often a black box. When something goes wrong, the debugging experience is close to reading tea leaves.

The eng-crew dashboard is a partial answer: every agent action, subtask assignment, approval event, and execution step is visible in real time. It's structured tracing for an engineering pipeline. But the field is far behind where distributed systems observability was in 2015, let alone 2025. We need agent-native OpenTelemetry. We don't have it yet.

08

Type Systems & Contracts→Structured Outputs & Schema Enforcement

A type system is a formal contract between a function and its callers. It says: this function accepts these shapes of data and returns that shape. Violations are caught at compile time, not at 3am when a production system sends a string where an integer was expected.

Language models produce freeform text. When an agent in a pipeline produces output that the next agent needs to parse, you have the equivalent of a dynamically typed interface between components — one that can fail at runtime in unexpected ways. Structured outputs (JSON schema enforcement, constrained generation) are type systems for agent interfaces. 

pseudo-schema
// Without schema enforcement (untyped interface)
architect_output = "Here's my plan: fix the auth flow, 
                    update the API, add some tests..."
// Downstream parser: good luck.

// With schema enforcement (typed interface)
architect_output = {
  "subtasks": [
    { "id": "auth-1", "description": "...", "agent": "coder", 
      "dependencies": [], "estimated_tokens": 2000 }
  ],
  "total_budget_usd": 1.20
}
// Downstream agent: valid, parseable, routable.

The eng-crew pipeline enforces structured output at every agent boundary. The architect produces a validated schema. The coder agents receive typed subtasks. The reviewer returns a typed evaluation. The contracts are explicit. The failures, when they happen, are predictable.

09

Dependency Management→Model Versioning & Prompt Stability

Dependency hell is what happens when your software relies on libraries that rely on other libraries, and upgrading one breaks another in ways that aren't immediately obvious. The solution — package locks, semantic versioning, reproducible builds — is about making the environment of a build explicit and reproducible.

AI pipelines have an equivalent: model versions, prompt versions, and the interaction between them. A prompt tuned for Claude Sonnet 3.7 may behave differently on Claude Sonnet 4. A system prompt that produces reliable structured output today may stop doing so after a silent model update. The "dependency" — the model — is opaque, continuously updated, and not under your control.

The emerging best practice is treating prompts as versioned artefacts, pinning model versions in production pipelines, and running eval suites on upgrade. It's dependency management for an environment where the dependency is a neural network. The tooling is embryonic. The need is the same age as software itself.

10

Build Systems & CI/CD→Agent Pipeline Orchestration & Evals

A good CI pipeline doesn't just run tests — it establishes what "correct" means for a codebase, and gates all changes against that standard. The combination of reproducible builds, automated tests, and deployment gates is what makes large-scale collaborative software development tractable.

Agentic pipelines need the same thing. Not just a sequence of agent calls, but a formal definition of correctness at each stage, automated checks before promotion, and rollback paths when an output fails. The architect → critic → approval gate → coder → reviewer → executor sequence in eng-crew is, structurally, a CI pipeline. The approval gate is a required review. The reviewer is an automated test suite. The executor is a deployment.

The human approval gate is not a limitation of autonomous AI. It is the CI gate — the deliberate, designed moment where human judgment is applied before an irreversible change propagates.

The Synthesis

What This All Points To

Across all ten of these analogies, the same pattern recurs: the AI engineering community is rediscovering, under different terminology, constraints that systems engineers have been navigating for decades. This isn't a criticism. Every technology generation has to earn its own understanding of the problem space. But it does suggest an opportunity.

The engineers who built garbage collectors, the researchers who designed Rust's borrow checker, the architects who created SQL parameterisation — they thought carefully about the failure modes of their systems at scale, over time, under adversarial conditions. Their solutions weren't intuitive; they were the product of seeing what happens when the naïve approach meets real production use.

We're at the naïve approach phase for AI agents. The production failures are starting to arrive. The next few years will produce the parameterised queries, the garbage collectors, the borrow checkers of agentic AI — formal solutions to the structural problems we keep stumbling into.

In the meantime, the fastest path to building AI systems that work reliably over time is to resist the amnesia. Ask what this problem is analogous to. Check whether it's been solved before. Build on that foundation rather than rediscovering it from first principles.

Memory needs a lifecycle. Shared state needs ownership semantics. Inputs need trust boundaries. Parallel agents need coordination models. Action scopes need enforcement. Systems need observability. Agent interfaces need contracts. These aren't AI insights. They're engineering insights. They apply here too.
